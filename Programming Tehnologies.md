# SDLC (Software Development Life Cycle)
 `Процесс создания программного обеспечения`
## Steps

### 1. Requirements Analysis (Анализ требований)
- **Цель** — собрать все требования к системе от заинтересованных сторон (клиентов, пользователей, менеджеров и т.д.) и документировать их.
- Для технической документации используется документ **SRS** (Software Requirements Specification), который содержит функциональные и нефункциональные требования к системе, а также требования к безопасности и производительности.
- `Примечание: SRS - Спецификация требований к программному обеспечению.`

### 2. Planning and Design (Планирование и дизайн)
- Производится планирование проекта, определяются задачи, сроки, ресурсы и бюджет.
- Оцениваются риски и разрабатываются стратегии их минимизации.
- Разрабатывается дизайн проекта, его системы, архитектурные решения.
- Определяются инструменты для разработки проекта: языки программирования, фреймворки, базы данных и другие инструменты, которые будут использованы при реализации.
- `Примечание: существует HLD (High Level Design) и LLD (Low Level Design).`

### 3. Implementation/Coding (Реализация/Кодирование)
- На этом этапе идет разработка проекта, пишется код в соответствии с техническими спецификациями.
- Соблюдаются стандарты написания кода; код должен быть легко читаемым, оптимизированным и легким в поддержке.

### 4. Testing (Тестирование)
- Проверка работоспособности и качества программного обеспечения.
- Тестирование отдельных модулей, их взаимодействия, тестирование всей системы в совокупности, а также производительности, безопасности и устранение ошибок.

### 5. Deployment (Развертывание/Выпуск)
- Доставка и установка программного обеспечения пользователям.
- Определение подходящей среды для развертывания и конфигурация на серверах.
- Обучение пользователей для эффективного использования программы.

### 6. Maintenance (Поддержка)
- Исправление ошибок и дефектов, оптимизация производительности, помощь пользователям и обновление программного обеспечения.


## SDLC Models

#### Waterfall (Каскадная модель):
   - Линейная и последовательная модель, где каждый этап выполняется один за другим.
   - Этапы: анализ требований → дизайн → разработка → тестирование → развертывание → сопровождение.
   - Переход к следующему этапу возможен только после завершения предыдущего.
   - Минус: сложно вносить изменения на поздних стадиях.

#### Agile (Гибкая модель):
   - Итеративный подход, где проект делится на небольшие циклы (итерации или спринты).
   - Постоянная обратная связь с заказчиком и возможность вносить изменения по ходу разработки.
   - Цель — быстро поставлять рабочие версии продукта и адаптироваться к изменениям.
   - Примеры методологий: Scrum, Kanban.

#### V-Model (V-образная модель):
   - Расширение Waterfall, где каждый этап разработки имеет свой этап тестирования.
   - Этапы разработки и тестирования расположены V-образно: например, анализ требований соответствует приёмочному тестированию.
   - Обеспечивает строгую связь между разработкой и тестированием.
   - Минус: так же, как и в Waterfall, сложно вносить изменения на поздних этапах.


Разница между V model и Agile:
В **Agile** разработка и тестирование идут параллельно в каждом спринте.

**Agile** для проектов с меняющимися требованиями, например, в разработке веб-приложений или мобильных приложений, где важно быстро адаптироваться к отзывам пользователей.

**V-Model** подходит для проектов с чёткими и неизменными требованиями, например, в медицинском или аэрокосмическом ПО, где ошибки недопустимы и нужно строгое тестирование.

## SRS (Software Requirements Specification)
`Документ описывающий требования к программному обеспечению.`
### Introduction (Введение)

##### 1. Purpose (Цель)
- Фиксация требований к проекту, описание продукта и ожидаемые результаты.
- Ясное описание продукта со всех сторон.

##### 2. Scope (Область применения)
- Описание границ системы, её функции и ограничения.
- Указание охваченных требований.

##### 3. Terminology/Definitions (Терминология/Определения)
- Содержит определения ключевых терминов и сокращений, используемых в документе.

### Overall Description (Общее описание)

##### 4. Main Features (Основные функции)
- Описание основных функций системы и её функциональности.

##### 5. User Classes (Классы пользователей)
- Определяет разные группы пользователей системы, их характеристики и потребности.

##### 6. Operating Environment (Операционная среда)
- Описание среды, в которой будет работать система (аппаратное и программное обеспечение, сетевые условия).

##### 7. Constraints (Ограничения)
- Указание ограничений, таких как - производительность, безопасность, законодательные и нормативные ограничения.

##### 8. Assumptions and Dependencies (Предположения и зависимости)
- Описание предположений, сделанных во время разработки и зависимости от других систем или технологий.

### For End

##### 9. Functional Requirements (Функциональные требования)
- Определение конкретных функций, которые система должна выполнять, включая входные и выходные данные.

##### 10. Non-Functional Requirements (Нефункциональные требования)
- Описание характеристик системы, таких как производительность, безопасность, надежность.

##### 11. System Models (Модели системы)
- Включает диаграммы случаев использования (UC-diagram), архитектуру системы и потоки данных.

##### 12. References (Ссылки)
- Перечисляет все источники и документы, использованные для составления SRS.

### ACID

- **A - Atomicity (Атомарность)**
- **C - Consistency (Согласованность)**
- **I - Isolation (Изоляция)**
- **D - Durability (Надежность)**

#### Atomicity (Атомарность)
- Гарантирует, что каждая транзакция будет выполнена полностью или не будет выполнена совсем.
#### Consistency (Согласованность)
- Каждая успешная транзакция фиксирует только допустимые результаты.
#### Isolation (Изоляция)
- Параллельные транзакции не должны оказывать влияние на результат друг друга.
#### Durability (Надежность)
- Гарантирует сохранение изменений после завершения транзакции, даже в случае сбоя системы.

### Isolation Levels
- **Read Uncommitted** (Чтение неподтвержденных данных)
- **Read Committed** (Чтение подтвержденных данных)
- **Repeatable Read** (Повторяемое чтение)
- **Serializable** (Сериализуемый): Полная изоляция транзакций.

## Database models:

- **Реляционные базы данных (RDBMS):** данные хранятся в таблицах с четкими связями между ними. Используют SQL для управления данными. Примеры: PostgreSQL, MySQL, Oracle. Подходят для систем, где важна целостность и структурированность данных.

- **Документные базы данных:** данные хранятся в виде документов (например, JSON). Более гибкие по структуре, хорошо подходят для неструктурированных данных. Примеры: MongoDB, CouchDB.

- **Ключ-значение:** данные хранятся в виде пар "ключ-значение", что делает их очень быстрыми для поиска по ключу. Часто используются в кэшировании и для простых структур данных. Примеры: Redis, Memcached.

- **Графовые базы данных:** данные организованы в виде узлов и связей между ними, что удобно для представления и работы с графами, например, в социальных сетях. Примеры: Neo4j, ArangoDB.

- **Колоночные базы данных:** данные хранятся в столбцах, что оптимизирует обработку больших объемов данных и аналитические запросы. Примеры: Apache Cassandra, HBase.
  
- **Векторные базы данных:** для хранения, обработки и поиска **векторов** — многомерных числовых представлений объектов (например, текста, изображений, аудио, видео).  Примеры: Pinecone, Weaviate, FAISS, Milvus, Qdrant (написана на Rust, популярна в open-source), Chroma

### DATA BASE NORMALIZATION

#### Виды нормализации форм:
- **1NF**
- **2NF**
- **3NF**

### **Первая нормальная форма (1NF)**

**Цель:** устранить многозначные атрибуты и привести структуру таблицы к плоской (табличной) форме.

1. Все столбцы содержат атомарные (неделимые) значения.
2. Каждый столбец должен содержать значения одного типа.
3. У каждой строки должен быть уникальный идентификатор (первичный ключ).

**Пример:**  
_До нормализации (не 1NF):_

| ID  | Имя  | Телефоны     |     |
| --- | ---- | ------------ | --- |
| 1   | Иван | 12345, 67890 |     |
| 2   | Анна | 54321        |     |

_После нормализации (1NF):_

| ID  | Имя  | Телефон |
| --- | ---- | ------- |
| 1   | Иван | 12345   |
| 1   | Иван | 67890   |
| 2   | Анна | 54321   |


### **Вторая нормальная форма (2NF)**

**Цель:** устранить частичные зависимости (зависимости не от всего первичного ключа).

1. Должна быть выполнена 1NF.
2. Каждый неключевой столбец должен зависеть от **всего** первичного ключа, а не от его части.

**Пример:**  
_До нормализации (не 2NF):_

| КурсID | Студент | Преподаватель |
| ------ | ------- | ------------- |
| 1      | Иван    | Петров        |
| 1      | Анна    | Петров        |

_После нормализации (2NF):_

 Таблица "Курс":  
 
| КурсID | Преподаватель |     |
| ------ | ------------- | --- |
| 1      | Петров        |     |

Таблица "Записи": 

| КурсID | Студент |     |
| ------ | ------- | --- |
| 1      | Иван    |     |
| 1      | Анна    |     |

### **Третья нормальная форма (3NF)**

**Цель:** устранить транзитивные зависимости (зависимости неключевых столбцов от других неключевых столбцов).

1. Должна быть выполнена 2NF.
2. Неключевые атрибуты не должны зависеть друг от друга.

**Пример:**  
_До нормализации (не 3NF):_

|СтудентID|Имя|Город|Почтовый индекс|
|---|---|---|---|
|1|Иван|Москва|101000|
|2|Анна|Санкт-Петербург|190000|

_После нормализации (3NF):_

- Таблица "Студенты":  

| СтудентID | Имя  | Почтовый индекс |     |
| --------- | ---- | --------------- | --- |
| 1         | Иван | 101000          |     |
| 2         | Анна | 190000          |     |

- Таблица "Города":  

| Почтовый индекс | Город           |     |
| --------------- | --------------- | --- |
| 101000          | Москва          |     |
| 190000          | Санкт-Петербург |     |

Эти формы нормализации помогают минимизировать дублирование данных и снизить риск логических аномалий при обновлении, удалении или вставке данных.

## **Stash**

**Stash** - это инструмент(хранилище) **Git** для временного сохранения изменений, которые не готовы для фиксации (commit)
**Stash** - пригодится когда:
- Нужно переключиться на другую ветку, не коммитя текущие изменения
- Изменения еще не закончены, но вы хотите временно их убрать, чтобы не мешали.
#### **Основные команды:**
```bash
	git stash - Сохранение изменений в Stash
	git stash list - Просмотр списка сохранений
	git stash apply -  Восстановление изменений
	git stash pop - Применение и удаление сохранения
	git stash clear -  Удаление всех сохранений
```

## **API**
**API (Application Programming Interface)** – это интерфейс для взаимодействия между программами. Он определяет, каким образом одна программа может запрашивать данные или функции у другой программы или сервиса.
### Основные типы API:

1. **REST API** – наиболее популярный тип API, основанный на HTTP-запросах (`GET`, `POST`, `PUT`, `DELETE`).
2. **SOAP API** – более сложный протокол, использующий XML.
3. **GraphQL API** – позволяет клиенту запрашивать только необходимые данные, улучшая производительность.
4. **WebSocket API** – обеспечивает двустороннюю связь в реальном времени.

### Где используется API?

- Взаимодействие веб-приложений с базой данных.
- Интеграция сервисов (например, использование карт Google в приложении).
- Разработка бэкенда и взаимодействие с фронтендом.
- Подключение сторонних платежных систем.

Простой пример **REST API** запроса:
```bash
curl -X GET "https://api.example.com/users/1"
```
### Основные HTTP-методы/CRUD команды:

| **Метод**  | **Описание**                 |
| ---------- | ---------------------------- |
| **GET**    | Получить данные              |
| **POST**   | Отправить новые данные       |
| **PUT**    | Обновить существующие данные |
| **PATCH**  | Частично обновляет ресурс    |
| **DELETE** | Удалить данные               |
### Структура REST API-запроса:

Пример: `https://api.example.com/users/1`

- `https://api.example.com` – **основной URL** сервиса.
- `/users/1` – **ресурс** (например, пользователь с ID = 1).
- Запрос можно дополнить **параметрами**: `?sort=asc&limit=10`.

### Testing types
1. **Smoke Test**
2. **Unit test**
3. **Integration test**
4. **Regression testing**
5. **Feature testing**
6. **Black Box testing**
7. **White Box testing**

---
1. **Smoke testing** — тестируется `работоспособность системы в целом`: запускается ли приложение и работают ли самые базовые функции. Это быстрая проверка сразу после сборки или деплоя, чтобы отсеять явно сломанные версии до начала полноценного тестирования.

2. **Unit testing** — тестируется `отдельная функция, метод или класс в изоляции от остальной системы`, часто с моками зависимостей. Аргумент простой: чем раньше найдена ошибка в конкретном кусочке кода, тем дешевле её исправить.

3. **Integration testing** — тестируется `взаимодействие между модулями`, например работает ли Flask-роут с реальной PostgreSQL. Юнит-тесты не гарантируют, что модули корректно работают вместе, поэтому нужна отдельная проверка именно стыков.

4. **Regression testing** — тестируется `старая функциональность после внесения новых изменений, чтобы убедиться, что ничего не сломалось`. Обычно автоматизировано и прогоняется в CI/CD при каждом коммите.

5. **Feature testing** — тестируется новая `функциональность целиком как единый сценарий, в соответствии с требованиями`. Например, вся цепочка восстановления пароля от запроса до смены.

6. **Black box testing** — тестируется поведение системы только по `входным и выходным данным, без знания кода внутри`. Показывает, соответствует ли система требованиям и документации, независимо от реализации.

7. **White box testing** — тестируется внутренняя логика кода: `конкретные ветвления, условия, пути выполнения`. Позволяет добиться высокого покрытия кода тестами и найти ошибки логики, невидимые снаружи.

---

8. **Acceptance testing (UAT)** — тестируется `соответствие готового продукта требованиям заказчика`; проводится конечными пользователями или их представителями, а не разработчиками.

9. **Load testing** — тестируется `поведение системы под ожидаемой рабочей нагрузкой`.

10. **Stress testing** — тестируется поведение системы под `экстремальной, предельной нагрузкой`, чтобы найти точку отказа.

11. **Performance testing** — тестируется `скорость отклика и пропускная способность системы`.

12. **End-to-End (E2E) testing** — тестируется `полный пользовательский сценарий через весь стек:` UI → backend → база данных. Инструменты: Selenium, Playwright, Cypress.

**Manual vs Automated testing** — ручное тестирование гибче для разовых и исследовательских проверок, автоматизированное (pytest, Jest, Selenium) экономит время на повторяющихся проверках вроде регрессии.

**TDD (Test-Driven Development)** — тест пишется раньше кода, затем пишется минимальный код, чтобы тест прошёл. Заставляет заранее чётко продумать требования к функции.

**Event-Driven Development (EDD)** — подход к архитектуре, где `компоненты системы общаются через события (events)`, а не через прямые вызовы друг друга: 
один компонент публикует событие ("заказ создан"), а другие подписчики на него реагируют независимо, не зная друг о друге напрямую. 
Это снижает `связанность (coupling) между сервисами и хорошо масштабируется` — типичный пример: очереди сообщений вроде RabbitMQ, Kafka, Redis Pub/Sub, где producer кидает событие в очередь, а один или несколько consumer'ов его обрабатывают асинхронно.

### Indexing tables in Database
Индекс — отдельная структура данных (обычно B-Tree), которая хранит ссылки на строки таблицы, отсортированные по значению колонки(ок). Ускоряет `SELECT`, но замедляет `INSERT/UPDATE/DELETE` (индекс тоже надо обновлять) и занимает доп. место на диске.

Без индекса БД делает **full table scan** — читает все строки, чтобы найти нужные. С индексом — быстрый поиск (аналог оглавления в книге вместо чтения всей книги).

#### Indexes types

1. **B-Tree (по умолчанию в PostgreSQL/MySQL)**
	- Универсальный, подходит для `"=, <, >, BETWEEN, ORDER BY"`
	- Самый частый выбор в 90% случаев

2. **Hash index** - Только для точного сравнения. Быстрее B-Tree в этом узком случае, но не поддерживает диапазоны/сортировку

3. **GIN (Generalized Inverted Index)** — PostgreSQL. Для составных типов: массивы, JSONB, полнотекстовый поиск

4. **GiST** - Для геометрических данных, полнотекстового поиска, диапазонов

5. **Composite (составной) индекс**
	- Индекс по нескольким колонкам сразу: `INDEX(user_id, created_at)`
	- Важно: порядок колонок имеет значение (left-most prefix rule)

6. **Unique index**
	- Гарантирует уникальность значений + попутно ускоряет поиск
	- Автоматически создаётся для `PRIMARY KEY` и `UNIQUE` constraint

7. **Partial index (частичный)** - Индекс только на подмножество строк по условию
	- Пример: `CREATE INDEX ON orders(status) WHERE status = 'pending'`

8. **Covering index** - Индекс содержит все колонки, нужные запросу → не надо ходить в саму таблицу (index-only scan)

### Когда индекс НЕ помогает / вредит
- На маленьких таблицах (full scan быстрее из-за оверхеда)
- На колонках с низкой селективностью (например, `is_active` — только true/false)
- Слишком много индексов → замедление записи (каждый INSERT/UPDATE обновляет все индексы)
- Использование функции над колонкой в WHERE без функционального индекса: `WHERE LOWER(email) = ...` не использует обычный индекс на `email`

### Пример на SQL (PostgreSQL)

```sql
-- Обычный индекс
CREATE INDEX idx_users_email ON users(email);

-- Составной индекс
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);

-- Частичный индекс
CREATE INDEX idx_orders_pending ON orders(status) WHERE status = 'pending';

-- Посмотреть, использует ли запрос индекс
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@test.com';
```

### Частые вопросы на собесе
- **"Как понять, что нужен индекс?"** — медленные запросы, `EXPLAIN ANALYZE` показывает `Seq Scan` вместо `Index Scan`
- **"Почему бы не проиндексировать вообще всё?"** — индекс замедляет запись и занимает место
- **"Разница между Clustered и Non-clustered index?"**
    - Clustered — физический порядок строк на диске = порядку индекса (обычно PRIMARY KEY, только один на таблицу)
    - Non-clustered — отдельная структура-ссылка на строки, можно создать несколько
